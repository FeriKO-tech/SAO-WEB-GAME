> Внутренний план. Не публикуется юзерам.
> Создан: 2026-04-19.
> Стартуем ПОСЛЕ полного завершения frontend портала (лендинг, auth, support backend, все полировки).

---

## 0. Жанр и референс

**Что делаем:** браузерный / мобильный **idle autobattle MMORPG** в стиле "Legend Online" / "Mushu" / `AFK Arena` / китайских клонов.

**Референс-скриншот** (обсуждали 2026-04-19):
- Классический 2D с пререндеренными 3D-спрайтами, изометрия сверху
- Combat Power (Мощь) как главная метрика
- VIP система, автобой, команда героев, крылья, экипировка
- Другие игроки видны в локации (soft-realtime, НЕ 60fps синхронизация)
- PvP асинхронный (сравнение снэпшотов на сервере)
- Много UI экранов (инвентарь, кузница, магазин, кланы, события)

**Что это НЕ:** не real-time MMO типа Tibia/WoW. Поэтому не нужен Colyseus/Socket.io для MVP.

## 1. Финальный стек (обсуждено, согласовано)

### Клиент (`packages/game/`)

| Слой | Тех | Зачем |
|------|-----|-------|
| Framework | React 19 + TypeScript | UI-тяжёлый проект, десятки экранов |
| Routing | TanStack Router | Типобезопасный роутер |
| Styling | Tailwind CSS + shadcn/ui | Быстрая верстка UI, готовые компоненты |
| UI state | Zustand | Герой, инвентарь, UI-флаги |
| Server state | TanStack Query | Кэш + invalidation для Firestore/Functions |
| Animations | Framer Motion | Переходы UI, модалки |
| 2D scene | Pixi.js v8 | Герои, монстры, карта, эффекты боя |
| Sprite anim | @pixi/spine (Spine 2D) | Анимация персонажей (стандарт жанра) |
| Audio | Howler.js | Фоновая музыка + SFX |
| Build | Vite 6 | Как и портал, единый инструмент |

Bundle target: ~800кб-1.2МБ gzipped (без ассетов).

### Сервер (serverless пока)

| Слой | Тех | Когда менять |
|------|-----|--------------|
| Auth | Firebase Auth | Уже есть, остаётся |
| DB | Firestore | Основное хранилище, достаточно до ~1000 DAU |
| Business logic | Cloud Functions (Node 20) | Боёвка, PvP, reward claim, валидация |
| Assets CDN | Firebase Storage или Cloudflare R2 | R2 выгоднее при росте трафика |
| Scheduler | Cloud Scheduler | Daily resets, события, очистка |

**Когда наберутся ~1000+ DAU:** мигрируем hot-path (чат локации, массовое PvP, real-time presence) на **Cloudflare Workers + Durable Objects** или **Node.js на Fly.io**.

**НЕ брать сейчас:** Colyseus, Socket.io, dedicated Node. Firebase-серверлесс покроет 100% MVP.

## 2. Архитектура репозитория (монорепо)

Текущий код переезжает в `packages/portal/`. Игра = новый пакет.

```
SAO-WEB-GAME/
├─ package.json                  <- корневой, workspaces
├─ firebase.json                 <- rewrites + functions config
├─ firestore.rules               <- общие правила
├─ packages/
│  ├─ shared/                    <- переиспользуемое
│  │  └─ src/
│  │     ├─ firebase.ts          <- initializeApp, getAuth, getFirestore
│  │     ├─ auth.ts              <- User type, signIn helpers
│  │     ├─ types/               <- User, Character, Hero, Item, Tick et
│  │     └─ i18n/                <- Language type, LanguageMap
│  ├─ portal/                    <- ТЕКУЩИЙ сайт, никакой логики не меняется
│  │  ├─ index.html
│  │  ├─ login.html
│  │  └─ vite.config.ts
│  ├─ game/                      <- новый
│  │  ├─ index.html
│  │  ├─ src/
│  │  │  ├─ main.tsx
│  │  │  ├─ App.tsx
│  │  │  ├─ scenes/              <- Pixi scenes (World, Battle)
│  │  │  ├─ ui/                  <- React components (HUD, modals)
│  │  │  ├─ state/               <- Zustand stores
│  │  │  └─ api/                 <- Cloud Functions callables
│  │  └─ vite.config.ts
│  └─ functions/                 <- Firebase Cloud Functions
│     ├─ src/
│     │  ├─ combat/              <- симуляция боя
│     │  ├─ pvp/                 <- async PvP engine
│     │  └─ rewards/             <- claim daily/event
│     └─ package.json
```

**Firebase Hosting rewrites** (`firebase.json`):
- `/` -> `packages/portal/dist/index.html`
- `/login`, `/support`, etc. -> соответствующий html из portal
- `/play/**` -> `packages/game/dist/index.html` (SPA fallback)

Портал и игра делят Firebase project: **один Auth = один логин работает везде**.

## 3. Пошаговый план разработки

**Шаги идут строго по порядку.** Каждый должен работать и быть задеплоен до следующего.

### Шаг 0: Монорепо (2-3 часа)
- [ ] Создать `packages/portal/`, переместить туда текущий код
- [ ] Создать `packages/shared/`, вынести `firebase.ts`, `auth.ts`, types, i18n-types
- [ ] Настроить npm workspaces в корневом `package.json`
- [ ] Проверить что `npm run build` портала работает
- [ ] Проверить что `firebase deploy` работает как раньше
- [ ] Обновить README

**Критерий:** текущий сайт после деплоя выглядит и работает идентично. Ни одного регресса.

### Шаг 1: Прототип "герой ходит по карте" (4-6 часов)
- [ ] `packages/game/` с React + Vite + Pixi v8
- [ ] Одна сцена, один спрайт героя (заглушка, цветной квадрат)
- [ ] Клик по карте -> герой идёт туда (простой pathfinding, пока прямая линия или A* на сетке)
- [ ] Камера следует за героем
- [ ] Firebase Auth гварда: если не залогинен -> redirect на `/login?returnTo=/play`
- [ ] Hosting rewrite `/play/**` -> `packages/game/dist/`

**Критерий:** логинимся на портале, жмём Play Now -> попадаем в игру, видим героя на карте, он двигается.

### Шаг 2: Первый бой (10-20 часов)
- [ ] 1 монстр на карте (статичный спрайт)
- [ ] Кликаем по монстру -> клиент шлёт `attackEnemy(enemyId)` в Cloud Function
- [ ] Cloud Function симулирует бой (детерминированный, `seed` возвращается)
- [ ] Ответ: список "кадров" боя (player HP, enemy HP, damage dealt)
- [ ] Клиент проигрывает анимацию по кадрам
- [ ] Монстр умирает -> выдаётся опыт (Firestore update)

**Критерий:** можно залогиниться, зайти в игру, убить монстра, увидеть что EXP вырос в Firestore.

### Шаг 3: Экипировка и статы (15-30 часов)
- [ ] Структура `characters/{uid}/` в Firestore: HP, ATK, DEF, EXP, Level
- [ ] Коллекция `inventory/{uid}/items/{itemId}`
- [ ] 3 слота экипировки: Weapon, Armor, Wings
- [ ] UI инвентаря (React + Tailwind): сетка, drag&drop или клик-надеть
- [ ] Combat Power пересчитывается из надетых предметов
- [ ] Отображение в HUD: Мощь, HP/MP бары, аватар

**Критерий:** находим предмет, надеваем, Combat Power вырос, урон по монстру вырос.

### Шаг 4-N: Остальные системы

| # | Система | Часов | Зависит от |
|---|---------|-------|-----------|
| 4 | Прокачка: level-up, stat points | 10-15 | 3 |

## 4. Критические решения, которые уже приняты

| Решение | Значение |
|---------|----------|
| Монорепо, не два репо | Shared код без копирования, один деплой |
| Firebase для всего бекенда в MVP | Serverless = дешево, не надо поднимать ноды |
| React + Pixi, НЕ только Pixi | UI-heavy игра, React сильно ускоряет разработку меню |
| Pixi v8, НЕ Phaser | Pixi = чистый рендерер, даёт больше контроля над архитектурой |
| Spine 2D для анимации | Индустрия-стандарт для 2D MMO-клонов |
| Tailwind + shadcn/ui | Десятки UI-экранов, нужен быстрый способ верстать |
| Zustand, НЕ Redux | Нет boilerplate, достаточно для сложной игры |
| TanStack Query, НЕ Apollo/SWR | Лучше интеграция с Firestore через мутации |
| Async PvP (снэпшоты), НЕ real-time | Укладывается в Cloud Functions, не нужен dedicated server |
| Ассеты на CDN, НЕ в репо | Гигабайты спрайтов нельзя в git |

## 5. Риски и что с ними делать

| Риск | Вероятность | Митигация |
|------|-------------|-----------|
| **Ассетов нет и негде взять** | Высокая | Купить набор на itch.io / Unity Asset Store / лицензировать Spine-пак. Без ассетов игра выглядит как прототип. |
| **Firestore чтения дорого при росте** | Средняя | Закешировать статику в CDN, использовать TanStack Query агрессивно. При >1000 DAU - мигрировать чат на отдельный realtime-слой. |
| **Читерство (модификация HP/урона)** | Высокая | **ВСЯ боевая логика и выдача наград только на сервере.** Клиент отображает результат. |
| **Cold start Cloud Functions** | Средняя | Использовать `minInstances: 1` для combat-функций при запуске живого сервера. |
| **PvP "нечестный" матчмейкинг** | Средняя | Матчить по Combat Power ± 20%. Позже - ELO-подобный рейтинг. |
| **Баланс экономики/донатов** | Высокая | Сначала проектировать экономику в Google Sheet, только потом писать код. Итеративно. |
| **Законодательство лутбоксов в РФ/ЕС** | Низкая | Избегать "чистых лутбоксов" - делать через гарантированные системы вроде pity-timer. |

## 6. Что НЕ делать в MVP (осознанный scope cut)

- 3D графика
- Голосовой чат
- Real-time совместные бои (co-op рейды) - только асинхронные
- Кастомизация персонажа (лицо, рост) - только стандартные классы
- Крафт с полной рецептурой - только "плавка" и "заточка"
- Трейд между игроками - только через аукцион (позже)
- Мобильные приложения (iOS/Android native) - пока только PWA
- Сюжетные кат-сцены - только текстовые диалоги
- Множественные сервера / шарды - один сервер, FCFS
